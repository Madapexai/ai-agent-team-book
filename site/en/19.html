External Tools — MCP and Toolchains

Where Are the Agents' "Hands"

The first time Yason realized Agents need tools was in a very awkward situation.

He said to Kai: "Help me check the production Nginx config and see why we're getting 502s."

Kai replied: "I don't have permission to access the server."

Then to Max: "Create a new project page in Notion for me."

Max replied: "I don't have the Notion API permission either."

Two Agents, two "sorry," both stuck on the same problem — they can talk, but they can't act.

An LLM without tools is just a talkative encyclopedia. An Agent with tools is a person who can actually work.

MCP: Model Context Protocol

For tool integration, Yason chose MCP (Model Context Protocol). MCP is an open protocol that defines how AI models interact with external tools and services.

The core architecture is simple:

Each tool runs in its own independent MCP Server. Agents call tools via the MCP protocol, rather than calling APIs directly.

Yason's MCP config looked like this:

{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/opt/data"]
    },
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_TOKEN": "${GITHUB_TOKEN}"
      }
    },
    "database": {
      "command": "python",
      "args": ["/opt/agents/mcp-servers/db-server.py"],
      "env": {
        "DB_URL": "${DB_URL}"
      }
    },
    "search": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-web-search"]
    },
    "feishu": {
      "command": "python",
      "args": ["/opt/agents/mcp-servers/feishu-server.py"]
    }
  }
}

After configuration, the Agent "grew hands" — it can read files, write code, query the database, send Feishu messages, and search the web.

A2A vs MCP: When to Use Which

As Yason's toolchain grew richer, he hit a new confusion: what exactly is the relationship between A2A and MCP? When should he use which?

MCP (Model Context Protocol): the protocol for Agents calling tools. Solves "how does an Agent operate external systems and data."

A2A (Agent-to-Agent): the protocol for Agent-to-Agent communication. Solves "how does an Agent delegate a task to another Agent."

Yason drew a comparison diagram:

# MCP scenario: Agent uses a tool
Agent ──MCP──→ Filesystem Server ──→ read/write files
Agent ──MCP──→ Database Server   ──→ query database
Agent ──MCP──→ GitHub Server     ──→ create PR

# A2A scenario: Agent asks another Agent for help
Agent A ──A2A──→ Agent B ──MCP──→ Database Server
                        └─MCP──→ Filesystem Server

Key differences:

Dimension MCP A2A
Communication target Agent → tool/service Agent → Agent
Message format Tool call (function call) Task delegation
Return result Tool output (data/file/status) Task result (deliverable/progress/status)
Suitable scenario File ops, API calls, DB queries Task decomposition, Sub-Agent management, cross-domain collaboration
Needs "understanding" No — tools execute by rules Yes — Agent must understand task intent

And the two can coexist. In Yason's Kai's daily work:

  1. Call the GitHub Server via MCP to create a PR
  2. Delegate the coding task to the OpenCode Sub-Agent via A2A
  3. The OpenCode Sub-Agent then calls the Filesystem Server via MCP to write code files

On a single task chain, MCP and A2A alternate, each doing its job.

MCP is the Agent's hands; A2A is the walkie-talkie between Agents. Hands can't make phone calls, walkie-talkies can't carry things — you need both.

MCP + RAG: The Knowledge Base Integration Pattern

After connecting MCP tools, Yason discovered a more powerful combination — MCP plus RAG (Retrieval-Augmented Generation).

Traditional MCP Servers use a "request-response" pattern: the Agent asks, the tool answers. But in many scenarios, what the Agent needs isn't real-time data from some API, but the documents and experience accumulated in a knowledge base.

Yason's solution was to add a RAG layer between MCP and the knowledge base:

Agent ──MCP──→ Knowledge Server ──RAG──→ Vector database
                              └──→ Experience card library
                              └──→ Technical doc library

Knowledge Server implementation:

@app.tool()
def query_knowledge(query: str, scope: str = "all"):
    """
    Query the knowledge base and return the most relevant content.
    scope: "technical" | "experience" | "all"
    """
    # Vectorize the query
    query_vector = embed(query)

    # Search different indexes based on scope
    if scope in ["technical", "all"]:
        docs = vector_db.search(query_vector, index="technical_docs", top_k=5)
    if scope in ["experience", "all"]:
        cards = vector_db.search(query_vector, index="experience_cards", top_k=5)

    return {
        "documents": docs,
        "experience_cards": cards
    }

This combo lets Kai automatically reference past experience while coding: first call the Knowledge Server via MCP, RAG retrieves relevant historical decisions and gotcha records, then start coding. Yason calculated that after adding this pattern, the probability of Kai stepping on the same rake dropped by about 80%.

MCP gives the Agent the ability to reach data; RAG gives the Agent the ability to understand data. Combined, the Agent goes from "an intern who looks things up" to "an experienced engineer."

This "safe-shell" tool is something Yason wrote himself — it lets the Agent run shell commands, but with safety restrictions:

# safe-shell.py - safe shell execution MCP Server
import subprocess, shlex
from mcp.server import Server

SAFE_COMMANDS = ["git", "npm", "python", "ls", "cat", "grep", "find"]
BLOCKED_PATTERNS = ["rm -rf", "dd if=", "> /dev/", "chmod 777"]

@app.tool()
def run_command(command: str, cwd: str = None):
    cmd_parts = shlex.split(command)

    if cmd_parts[0] not in SAFE_COMMANDS:
        return {"error": f"Command {cmd_parts[0]} is not in the allow list"}

    for pattern in BLOCKED_PATTERNS:
        if pattern in command:
            return {"error": f"Command contains a forbidden pattern: {pattern}"}

    result = subprocess.run(
        command, shell=True, cwd=cwd, 
        capture_output=True, text=True, timeout=30
    )
    return {
        "stdout": result.stdout[-2000:],
        "stderr": result.stderr[-1000:],
        "exit_code": result.returncode
    }

The Tool Permission Model

The more tools, the bigger the risk. Yason quickly faced permission-management issues — not every tool should be open to every Agent.

For example: Kai needs to write code, but he shouldn't be able to directly modify the production database. Max needs Feishu access, but he shouldn't be able to delete the repo.

Yason designed a three-tier permission model:

Permission tiers:
  Level 0: Read-only tools
    - search, read-file, ls, git log
    - granted to all Agents by default
    Risk: very low

  Level 1: Write-type tools (non-destructive)
    - write-file (limited to workdir), git commit, feishu send
    - requires Yason's authorization within the task
    Risk: low-medium

  Level 2: High-risk tools
    - shell (unrestricted mode), database write, git push (prod branch)
    - each use requires Yason's individual confirmation
    Risk: high

Concrete implementation:

# /opt/agents/permissions.yaml
agents:
  kai:
    default_level: 1
    tools:
      filesystem: { level: 1, paths: ["/code/repos/*"] }
      github: { level: 1, branches: ["feature/*", "fix/*"] }
      terminal: { level: 1, commands: ["git", "npm", "python"] }
      database: { level: 2  # writing to DB needs confirmation }

  max:
    default_level: 0
    tools:
      search: { level: 0 }
      feishu: { level: 1, actions: ["send_message", "create_doc"] }
      browser: { level: 0 }

The Principle of Least Privilege

As tools grew, Yason distilled a core security principle: each Agent should only have the tools it needs to complete its task.

In practice, this means:

  1. Group by role: the Dev Agent only gets dev tools, the Ops Agent only gets ops tools — no overlap
  2. Runtime permission downgrade: during off-hours (e.g., an early-morning auto post-mortem), the Agent can only use read-only tools
  3. Tool-level allowlist: not "can't access sensitive tools," but "by default can only access tools on the allowlist"

After months of iteration, each Agent's MCP tool list in Yason's permission config is carefully trimmed. Kai has no Feishu-send permission, Max has no shell-exec permission, Rex has no GitHub-write permission — every permission that's missing is one fewer incident point.

Tool permission management isn't restricting the Agents — it's protecting yourself. The more tools, the more finely you must tune permissions. Yason's rule is simple: an Agent's number of tools is inversely proportional to its permission tier — lower tier means more tools; higher tier means fewer tools.

The Painful Lesson of the Security Line

Yason's awareness of tool security came from a real incident.

An Agent (name withheld) was debugging a database issue and directly ran this SQL:

DELETE FROM users WHERE last_login < '2024-01-01';

The Agent meant to clean up test data, but it was connected to the production database.

Fortunately, a layer of protection was in place — Yason's MCP Server checks the table name and where-clause before any write-type SQL:

@app.tool()
def execute_query(sql: str):
    # Safety check
    sql_lower = sql.lower().strip()
    dangerous_patterns = ["drop", "truncate", "delete", "update"]

    for pattern in dangerous_patterns:
        if sql_lower.startswith(pattern):
            if not confirm_with_yason(sql):
                return {"error": "High-risk SQL blocked; Yason notified"}

    return db.execute(sql)

After this incident, Yason added a hard rule: any high-risk operation must first send a Feishu card to Yason for confirmation before it can execute.

Agent → Yason: ⚠️ High-risk operation confirmation
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Operation: DELETE FROM users
Rows affected: estimated 15,342
Environment: production database (prod-db-01)
┌──────────────────────────────┐
│  Confirm execution │ Reject  │
└──────────────────────────────┘

Yason glances at the card on his phone, taps "Confirm" or "Reject" — the whole thing takes 3 seconds. This confirmation mechanism adds one step, but since then, there have been no more database mishaps.

Custom Tools: Let Agents Do What You Want

The MCP ecosystem has plenty of ready-made tools, but Yason found that the real efficiency gains came from a few custom little tools.

For example, he wrote a "task dependency analyzer":

@app.tool()
def analyze_dependencies(task_description: str):
    """
    Analyze a task description and identify all dependencies and prerequisites.
    """
    prompt = f"""
    Analyze the dependencies of the following task:
    Task: {task_description}

    Output JSON format:
    {{
        "dependencies": ["prerequisite task A", "prerequisite task B"],
        "requires_approval": true/false,
        "estimated_complexity": "low" | "medium" | "high",
        "suggested_split": ["subtask 1", "subtask 2"]
    }}
    """
    response = llm.call(prompt)
    return parse_json(response)

This tool lets Kai do a "feasibility analysis" before taking a task, instead of blindly diving in. Yason later calculated that after adding this tool, the mid-task rework rate dropped by 60%.

The ultimate goal of the toolchain isn't to let Agents do more — it's to make Agents "think clearly" before acting. Tools are the Agent's extension, not its controller.

Chapter Summary

  • The toolchain is the key to Agents going from "chatting" to "working" — but the more tools, the higher the management cost
  • MCP is the vertical Agent→tool protocol; A2A is the horizontal Agent↔Agent protocol — they complement each other
  • The MCP ecosystem is mature — common services like GitHub, Slack, Notion, and Postgres all have open-source MCP Servers
  • Least-privilege principle: each Agent should only have the minimal tool set needed to complete its task
