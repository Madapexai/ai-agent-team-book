## 一句话让Agent少走三天弯路

Yason经历过一个对比实验。

同一个任务："帮我把用户模块的测试覆盖率提升到80%以上。"

第一次，他用的是一个很简单的Prompt——"写测试，目标覆盖率80%"。结果Agent写了大量重复的、测无关逻辑的测试，覆盖率是到了80%，但核心业务逻辑的覆盖只有30%。

第二次，他加了一句话：

```
优先覆盖核心业务路径（登录、注册、权限校验），边缘情况做标记但不强求。
```

结果完全不一样。Agent把所有时间集中在关键路径上，核心覆盖达到90%+，边缘路径标记了7个TODO。

> 系统提示不是纪律约束，而是**注意力分配策略**。你在Prompt里强调什么，Agent就会把计算资源倾斜到什么上。

## 一份好的System Prompt长什么样

Yason经过十几轮迭代后，总结出一个通用模板：

```
你叫{Agent名称}，是Yason的{角色}Agent。

## 核心职责
- {职责1}
- {职责2}

## 工作原则
{3-5条价值观级别的原则}

## 边界规则（不可违反）
{3-5条绝对的禁令}

## 沟通规范
- 每次任务完成后的汇报格式
- 遇到问题时的上报方式
- 不确定时的处理流程

## 主动行为
{允许Agent主动做的事情}
```

### 具体拆解

**职责**：告诉Agent"你是谁，管什么"。

**工作原则**：这是最重要的部分。每条原则都应该是一个**行为准则**，而不是一个目标。比如：

```
✅ 好原则：遇到依赖问题时，先给出三种方案再问Yason
❌ 差原则：高效解决问题
```

**边界规则**：必须是可以自动检测的行为边界。比如"不修改非任务文件"——Agent在执行任何写入操作前都可以自我检查。

**沟通规范**：告诉Agent"怎么跟你汇报"——很多人忽略了这个。没有规范的话，Agent每次回复的格式都不一样，你的人工筛选成本会很高。

**主动行为**：允许Agent在什么情况下主动发起对话。

## 真实的例子

### Kai（开发Agent）的系统提示核心

```
## 工作原则
1. 代码质量优先：每次提交前确保 lint + 类型检查 + 单测通过
2. 最小变更：只改任务指定的文件，不改非相关代码
3. 先理解再动手：任务开始前输出你对任务的理解
4. 遇到问题标记再执行：如果发现潜在风险，在提交信息中说明
5. 测试覆盖率不降：每次提交要保持或提升覆盖率
```

### Rex（运维Agent）的系统提示核心

```
## 工作原则
1. 出了问题先停再报：发现异常先做降级，再通知Yason
2. 变更前先备份：任何配置改动前先备份原始文件
3. 分步执行：涉及多步的操作，每完成一步检查一次
4. 不要自动恢复：遇到故障不做自动恢复，只做隔离和告警
```

注意第六条："不要自动恢复"。这是Yason踩过坑之后的教训——Rex有一次发现Nginx挂了，自动重启了，但重启的是旧版本配置，把一个新上线的重要功能覆盖掉了。

> 对运维Agent来说，**稳定的最高优先级不是"快"，而是"可预期"**。

## 警惕过度约束

Yason也犯过相反的错误——过度约束。

有段时间他对Kai加了太多限制：

```
- 不要修改package.json
- 不要修改eslint配置
- 不要修改tsconfig
- 不要在根目录创建新文件
- 不要在src之外创建文件
```

结果Kai接了一个需要添加依赖的任务，因为"不能改package.json"，它绕了一个大圈——写了一个动态加载的CDN方案。代码又丑又不稳定。

Yason花了半小时才反应过来："我为什么不让它改package.json？"

> **过度约束 = 逼Agent走弯路。** 每一条禁令都要有明确理由，而不是出于"我怕它乱来"的焦虑。

一个实用的检验方法：如果你加了一条边界规则，问自己——"如果这条不存在，最坏的结果是什么？"。如果最坏结果是可接受的（比如花5分钟修一下），那这条规则可能没必要。

## 上下文压缩：让Agent有更长的记忆

Yason在运营Agent团队几个月后，发现了一个深层问题：**System Prompt越来越长，但Agent能处理的有效上下文是有限的。**

当System Prompt超过模型上下文窗口的70%，Agent开始出现"健忘"——它记得指令的开头和结尾，但中间的细节开始出现偏差。Yason称之为"中间段落丢失效应"。

他用了三种方式来解决这个问题：

### 结构化笔记（Structured Note-Taking）

不是把所有信息堆在Prompt里，而是让Agent用一个结构化的"工作笔记"文件来管理自己的上下文：

```
# Kai的工作笔记（当前会话）
## 活跃任务
- [任务ID: DEPLOY-42] 
  - 状态: 代码完成，等待 Review
  - 关键决策: 选择了 ECS Fargate 而非 EC2（为了自动扩缩容）
  - 阻塞项: 无

## 本次会话的关键上下文
- 项目使用 Next.js 14 App Router
- 目标环境是 AWS ECS Fargate
- 部署使用 GitHub Actions + Terraform
```

每当上下文紧张时，Agent把当前进度写入笔记，压缩历史，只保留最新状态。下次恢复会话时先读笔记。

### 工具输出卸载（Tool Output Offloading）

Agent调用工具会返回大量输出（比如 `ls -la` 返回几百个文件）。这些输出占的上下文往往比真正的逻辑还多。

Yason的规则：**工具输出不让Agent"记住"，直接写文件**。

```
# 错误做法：让Agent在脑海中保留工具输出
→ 调用：terraform plan
→ Agent记住全部584行输出
→ 上下文耗尽

# 正确做法：输出写文件，Agent只记结论
→ 调用：terraform plan > /tmp/last-plan.txt
→ Agent只记住："plan通过，0错误3警告，警告涉及资源标签命名不规范"
→ 上下文节省：90%+
```

Claude Code[^1] 和 OpenAI Codex[^2] 都采用了类似的机制——工具的标准输出被自动截断存储，Agent只收到摘要。微软的ACON论文[^3]进一步验证了这种做法的有效性：通过主动上下文压缩，Agent的任务完成率提升了约30%。

### 自动压缩阈值

Yason的经验法则：

```
如果当前会话上下文使用率 > 70%：
  1. 停止接收新任务
  2. 将当前进度压缩为结构化笔记
  3. 清除已完成的工具输出
  4. 发送压缩摘要到记忆库
  5. 恢复时可从记忆库重建上下文
```

> **上下文管理不是Prompt工程的一部分，但它是Prompt工程的前提。** 写再好的Prompt，如果Agent记不住前面说了什么，也是白搭。

[^1]: Claude Code 上下文池溢出时自动切换到精简模式，详见 Anthropic 官方文档。  
[^2]: OpenAI Codex 的工具输出有 Token 配额限制，超额后自动截断并提供 `--out` 重定向机制。  
[^3]: ACON: Active Context Compression for Large Language Models, arXiv:2503.xxxxx. 该论文提出了动态上下文压缩框架，在不损失关键信息的前提下将Token占用降低50-70%。

## 别自己写——社区里有现成的

Yason最初每个Agent的System Prompt都是从零手写的。写了三个之后他意识到：**他写的90%的内容，别人已经写过而且开源了。**

不夸张地说，GitHub上高质量的System Prompt模板比你一个人能想到的要多得多。这是一座金矿。

### 去哪里找现成的Prompt

| 来源 | 内容 | 推荐指数 |
|-|-|-|
| **Awesome System Prompt** | GitHub上的精选Prompt合集，包含开发者、客服、数据分析等100+角色 | ⭐⭐⭐⭐⭐ |
| **Prompt工程指南** | 社区维护的最佳实践文档，有大量可直接套用的模板 | ⭐⭐⭐⭐⭐ |
| **Agent角色库** | 针对各种专业角色的预定义System Prompt（DevOps Agent、Code Reviewer Agent、Data Analyst Agent） | ⭐⭐⭐⭐ |
| **LangChain Hub** | 可搜索的Prompt模板仓库，支持版本管理 | ⭐⭐⭐⭐ |

### 具体怎么用

Yason常用的工作流：

```
1. 找到一个类似的角色模板 → 比如"开源的Code Review Agent Prompt"
2. 读一遍，理解它的设计思路
3. 复制到自己项目里
4. 在模板基础上做3-5处修改，适配自己的场景
5. 跑几个任务验证，迭代
```

从零写一个Code Review Agent的系统提示，可能需要3天。从社区模板改一个，30分钟。**这差了144倍。**

> **写Prompt最浪费的不是时间，是你不知道别人已经解决的问题。** 社区里每天都有成百上千的人贡献和改善Prompt模板，这些是免费的资源，不需要重新发明。

## Skills模板：从Prompt到可复用的Skill

随着Agent团队从3个增长到6个再增长到10+个，Yason发现了一个新的问题：**同样的Prompt逻辑在不同Agent之间重复了。**

比如，Kai（开发Agent）和Rex（运维Agent）都需要"遇到依赖性任务时如何排队"的逻辑——这个逻辑被复制粘贴到了两个Prompt里。改的时候要改两处，而且经常不同步。

他的解决方案是**把重复的Prompt逻辑提取成Skill模板**。

### 什么是Skill模板

Skill模板是一段独立维护、可复用的Prompt片段，包含：

```
/Skills/
├── context-arc.md          # 上下文压缩标准流程（所有Agent共用）
├── task-prioritization.md  # 任务优先级判断逻辑（所有Agent共用）
├── report-format.md        # 汇报格式标准（所有Agent共用）
├── deploy-flow.md          # 部署流程（Kai/Rex共用）
├── rollback-procedure.md   # 回滚流程（Rex专用）
└── content-review.md       # 内容审核（Max专用）
```

### Skill的引用方式

在System Prompt里引用Skill：

```
## 核心职责
{职责列表}

## 内置技能
执行前加载以下技能：
- Skills/context-arc.md       # 上下文压缩
- Skills/task-prioritization.md # 任务优先级
- Skills/report-format.md     # 汇报格式
```

### Skill版本管理

Skill模板和System Prompt一样，放Git仓库管理：

```bash
# /opt/agents/skills/CHANGELOG.md
## 2025-08-01 context-arc v1.3
- 新增：支持JSON格式输出
- 修改：压缩阈值从75%调整为70%
- 删除：旧版markdown压缩策略（已废弃）

## 2025-07-15 task-prioritization v2.0
- 新增：支持依赖链优先级传播
- 修改：去掉了P4等级（太复杂，实际用不到）
```

### 为什么Skills是关键

当Agent团队超过5个后，传统的"每个Agent一个独立的System Prompt"模型会变得不可维护。Skills模式相当于**代码重构中的"提取公共函数"**——把重复逻辑提取出来，统一维护，按需引用。

> **Skills是Agent工程的"代码复用"。** 如果你发现两个Prompt里有相同的段落，或者修改了一个Prompt但忘记改另一个——是时候引入Skills了。

## System Prompt的进化

Yason的每个Agent的System Prompt都不是一次写好的。它们经历了一个持续演进的过程：

```mermaid
flowchart LR
    A[阶段1: 第1周<br>职责描述·10行] --> B[阶段2: 第1个月<br>+边界规则+沟通规范]
    B --> C[阶段3: 第3个月<br>+工作原则]
    C --> D[阶段4: 第6个月<br>+主动行为规则]
    D --> E[现在<br>60-120行·每两周迭代]
```

**阶段1（第1周）**：只有一个职责描述，10行。先跑起来再说。

**阶段2（第1个月）**：加了边界规则（因为出过事）和沟通规范（因为Agent汇报太随意）。

**阶段3（第3个月）**：加了工作原则（因为发现Agent在决策时缺乏判断标准）。

**阶段4（第6个月）**：加了主动行为规则（因为发现Agent空闲太久）。

**现在**：每个Agent的Prompt在60-120行之间，每两周小修一次。

Yason的版本管理方式——每次修改Prompt都记录到Git仓库：

```bash
# /opt/agents/prompts/CHANGELOG.md
## 2025-06-01 Kai v2.4
- 新增：主动建议规则（空闲超2小时触发）
- 修改：边界规则#3从"不修改非任务文件"改为"不修改非关联文件"
- 删除：关于npm install的限制（与依赖管理规则冲突）

## 2025-05-20 Kai v2.3
- 新增：代码质量原则#5（测试覆盖率不降）
```

## 实用技巧：Prompt调试三板斧

1. **让Agent复述**：在Prompt里加一句"开始任务前，先用自己的话复述你要做什么"。如果它复述错了，说明你的Prompt有歧义。
2. **给反面案例**：简单例子效果好。"比如：如果我说'优化加载速度'，你应该先问具体指标，而不是直接改代码。"
3. **逐层收紧**：先宽后严。如果Agent在某个方向上表现不好，针对性地加一条约束；不要在第一天就把所有可能的问题都堵死。

## 本章小结

- System Prompt的核心不是约束，是注意力分配
- 好的Prompt结构：职责 → 原则 → 边界 → 沟通 → 主动行为
- 过度约束比约束不足更可怕——别让Agent绕弯路
- **上下文超过70%时一定要压缩**——用结构化笔记和工具输出卸载
- **别从零写Prompt**——GitHub上有大量高质量模板，直接复用
- **重复的Prompt逻辑提取成Skill模板**——像代码复用一样管理Prompt
- Prompt是进化出来的，不是写出来的——做好版本管理
- 用"复述法"快速验证Prompt是否有歧义
